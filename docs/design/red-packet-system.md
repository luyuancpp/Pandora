# 红包服务器设计备忘

> 目标:记录"秒杀系统"到"红包系统"的服务端设计映射,后续如果实现红包玩法 / 活动红包 / 公会红包,以本文作为初始设计入口。

## 1. 秒杀系统到红包系统的映射

红包系统和秒杀系统的核心相似点是:大量用户在短时间内争抢有限资源,服务端必须保证扣减原子性、防重复、防刷、异步落库和最终可对账。

| 秒杀系统 | 红包系统 |
|---|---|
| 商品库存 | 红包剩余个数 / 剩余金额 |
| 一人一单 | 一人只能抢一次 |
| 下单记录 | 领取记录 |
| 扣库存 | 扣红包份额 |
| MQ 异步创建订单 | MQ 异步写领取流水 |
| 防刷 / 限流 | 防止脚本抢红包 |
| 秒杀结束 | 红包抢完 / 过期退款 |

## 2. 服务端核心流程

1. 发红包:创建红包批次,写入 MySQL 权威记录,预拆金额或记录总金额与剩余份数。
2. 初始化缓存:把红包剩余份数、剩余金额、已领取用户集合写入 Redis。
3. 抢红包:入口先做登录态校验、限流、防重复领取校验。
4. 原子扣减:使用 Redis Lua 一次性完成"是否已抢过 -> 是否还有剩余 -> 扣份数/金额 -> 记录领取标记"。
5. 异步落库:扣减成功后投递 Kafka / MQ,由消费者写领取流水、资金流水、审计日志。
6. 结果返回:抢到则返回金额和领取记录 id,没抢到则返回已抢完 / 已领取 / 已过期。
7. 过期退款:红包到期后扫描未抢完金额,生成退款流水并关闭红包批次。

## 3. 必须保证的不变量

- 同一个玩家对同一个红包只能领取一次。
- 红包发出总金额 = 已领取金额 + 已退款金额。
- 红包剩余份数不能小于 0。
- 红包剩余金额不能小于 0。
- Redis 扣减成功但 MQ / DB 失败时,必须能通过补偿任务恢复领取流水。
- MQ 至少一次投递时,落库必须靠 idempotency_key 幂等。
- 金额统一用整数"分"存储,禁止使用浮点数。

## 4. 建议的数据与中间件分工

| 组件 | 职责 |
|---|---|
| Redis | 高并发抢红包入口、库存/份数原子扣减、领取去重 |
| MySQL | 红包批次、领取记录、资金流水、退款记录、审计 |
| Kafka / MQ | 异步落库、削峰、补偿事件、审计事件 |
| 限流组件 | 按玩家 / IP / 设备限制请求频率,降低脚本压力 |
| 风控模块 | 黑名单、异常设备、异常频率、批量账号识别 |

## 5. 推荐 Redis Key 草案

按 `infra.md` 的命名规范,红包域统一放在 `pandora:redpacket` 下:

| Key | 类型 | 用途 |
|---|---|---|
| `pandora:redpacket:<packet_id>` | hash | 红包批次状态,如 total_amount、left_amount、left_count、status |
| `pandora:redpacket:claimed:<packet_id>` | set | 已领取玩家集合 |
| `pandora:redpacket:amounts:<packet_id>` | list | 预拆好的每份红包金额 |
| `pandora:redpacket:lock:<packet_id>` | string | 过期退款 / 补偿任务互斥锁 |

Redis Cluster 下,同一个红包的多个 key 如果要在一段 Lua 内同时访问,需要使用 hash tag,例如:

```text
pandora:redpacket:{<packet_id>}:meta
pandora:redpacket:{<packet_id>}:claimed
pandora:redpacket:{<packet_id>}:amounts
```

## 6. 初始表设计方向

| 表 | 用途 |
|---|---|
| `redpacket_batches` | 红包批次:发送人、总金额、总份数、剩余金额、剩余份数、状态、过期时间 |
| `redpacket_claims` | 领取记录:红包 id、领取人、金额、领取时间、幂等键 |
| `redpacket_refunds` | 退款记录:红包 id、发送人、退款金额、退款时间、幂等键 |
| `redpacket_ledger` | 资金流水:发出、领取、退款、补偿、审计 |

## 7. 和秒杀系统不同的地方

- 秒杀扣的是商品库存,红包扣的是金额和份数,还涉及资金安全。
- 秒杀失败通常只是无库存,红包还要处理过期退款。
- 红包必须可对账:发出、领取、退款、平台补偿都要有流水。
- 红包金额拆分要保证公平性和边界条件,比如最小 1 分、总额不超发。
- 红包入口要有更强风控,防脚本抢、批量注册、设备农场。

## 8. 参考实现方向

可以参考秒杀系统的高并发思路,但不能直接照搬为生产红包系统:

- Redis Lua 做原子扣减。
- MQ 做异步削峰。
- MySQL 做最终权威账本。
- 领取流水使用幂等键防重复消费。
- 定时任务处理过期退款和 Redis / MySQL 差异补偿。

生产实现前必须补齐压测、账务对账、安全审计和风控策略。
