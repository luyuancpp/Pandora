# Pandora DDD 架构口径

> 记录 Pandora 对 DDD、微服务、事件驱动的取舍。本文是 `pandora-arch.md` §3.1 的展开版。

## 1. 结论

DDD 实用,但不是银弹。它更像一套"管理复杂业务的建模方法",不是某种固定架构模板。

一句话结论:

> DDD 适合复杂业务边界清晰、规则多、变化多的系统;不适合简单 CRUD、早期原型、性能主链路里无脑套。

Pandora 会吸收 DDD 的边界、聚合、领域事件和事务边界思想,但不把代码写成重型教科书 DDD。

## 2. 微服务 + 事件不等于 DDD

这是最容易误解的地方。微服务、事件、DDD 三者关系如下:

| 概念 | 本质 |
|---|---|
| DDD | 怎么理解业务、划分边界、建模规则 |
| 微服务 | 一种部署和运行架构 |
| 事件 | 一种模块 / 服务之间通信方式 |

DDD 关注的是:

- 这个业务里有哪些核心概念?
- 玩家、账号、角色、背包、订单、匹配、房间、战斗分别属于哪个边界?
- 哪些规则必须强一致?
- 哪些变化可以异步最终一致?
- 一个聚合里什么能一起改,什么不能一起改?
- 模块之间应该通过接口还是事件协作?

微服务只是其中一种可能落地方式。Pandora 可以接受以下组合:

- DDD + 模块化单体
- DDD + 微服务
- 不用 DDD 的微服务
- 事件驱动但不是 DDD

很多"微服务 + MQ + 事件"系统其实只是分布式 CRUD:服务拆了,事件发了,但边界没想清楚,最后会变成跨服务互相调用、数据到处复制、问题更难排查。

## 3. Pandora 是否会朝 DDD 方向发展

如果 Pandora / XuanMing-Server 业务继续变复杂,会**部分朝 DDD 的思想发展**,但不变成"教科书 DDD 架构"。

更现实的方向:

- 账号、角色、背包、交易、匹配、房间、战斗、赛季、排行榜这些模块边界越来越清楚。
- 核心业务使用明确的领域模型,而不是所有逻辑堆在 handler / service 里。
- 重要状态变化发布领域事件 / 业务事件。
- 强一致的东西收在同一个事务边界里。
- 异步事件用于通知、统计、日志、任务、排行榜等旁路逻辑。
- 先做模块化边界或少量服务,等瓶颈明确后再拆更细的微服务。

不建议一开始就"全量 DDD 化"。业务复杂度没起来时,重型 DDD 会把开发速度拖慢,还会制造过早抽象。

## 4. 游戏服务器里的适用边界

游戏服务器不是所有地方都适合重 DDD。

不适合重 DDD 的部分:

- 战斗帧同步 / 状态同步:更偏性能、状态机、协议、仿真。
- 网关连接层:更偏网络和会话管理。
- Redis 缓存 / 排行榜热路径:更偏数据结构和吞吐。
- 简单配置读取:不需要 DDD。

适合借鉴 DDD 边界意识的部分:

- 交易 / 背包 / 订单 / 资产:适合更严肃的领域建模、聚合边界、幂等键和本地事务 / outbox。
- 匹配 / 房间 / 队伍:适合建模和状态机,但别过度抽象。
- 战斗:适合状态机 / 组件化 / 规则引擎思路,DDD 只做外围业务边界。
- 日志 / 监控 / 运维:不需要 DDD,保持工具化和可观测性优先。

## 5. Pandora 的落地路线

务实路线:

1. 先保持模块化边界优先,不要过早微服务化。
2. 每个业务模块有自己的 model / service / repository / event 边界。
3. 跨模块不要随便直接读写对方数据,先通过接口或事件。
4. 交易、背包、资产这类高风险模块,使用更强的聚合 / 事务边界。
5. 事件先用于旁路和最终一致,不要一上来把核心链路全拆成事件流。
6. 真要拆微服务时,按已经稳定的业务边界拆,而不是按技术层拆。

## 6. 项目硬口径

DDD 不是"微服务 + 事件",而是决定哪些东西应该成为服务、哪些东西应该发事件、哪些东西必须留在一个事务里的思考框架。

Pandora 后续代码评审按这个口径判断:

- 不因为用了 Kafka / MQ 就声称是 DDD。
- 不因为拆成多个服务就声称边界合理。
- 不为了 DDD 名词制造 `FactoryManager` / `DomainServiceFactory` 这类空壳抽象。
- 强一致资产链路优先看事务边界、幂等键、补偿、outbox,而不是看分层名词。
- 性能主链路优先看 tick 预算、状态机、数据结构、协议开销,DDD 只作为外围边界参考。
